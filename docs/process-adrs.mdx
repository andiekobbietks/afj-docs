## What is an ADR

An Architecture Decision Record is a short note that captures one decision, not a formal sign-off, not a long report, just a record anyone can read later and understand why something was built the way it was. The format below follows the UK Government Digital Service's own ADR framework, introduced by their Enterprise Architecture team to help teams across government make and record decisions consistently.

**Why it matters to non-engineers, too**

*For engineering*

Stops the same debate happening twice. When someone asks "why don't we just use Tailwind," the ADR already has the answer, with the alternatives that were weighed and rejected on record.

*For marketing / brand*

Explains why a visual choice (like the wine-to-gold gradient) is locked in as policy, not a developer's personal preference, so brand reviews reference the decision instead of relitigating the colour.

*For anyone joining later, including Plamedi*

Ramp-up without asking around. Reading four short paragraphs (what was the situation, what were the choices, what got picked, why) explains a decision faster than a meeting would, and the reasoning is written down instead of living in one person's head or a deleted Slack thread.

## ADR: Bootstrap as the theming framework

The actual ADR behind why every component on this page is built on Bootstrap 5.3 rather than a bespoke CSS system or Tailwind, following the context → options → decision → reasoning structure.

<AdrCard
  title={"Adopt Bootstrap 5.3 as the base component framework, themed via CSS custom properties"}
  status={"Accepted"}
  context={"AFJ Cardiff's original site used hand-written CSS classes ( .afj-btn-hero , .afj-feature-card , etc.) styled with hardcoded hex values. Every new page meant re-solving layout, responsive behaviour, and accessibility from scratch. The commerce build (product grid, cart, checkout) needed a faster, more consistent base without losing the brand's specific look."}
  options={"1. Keep hand-rolled CSS , full control, but every grid, gap, and responsive rule gets rebuilt per page. 2. Adopt Tailwind , fast utility authoring, but a compiler step and a different mental model from the team's existing CSS. 3. Adopt Bootstrap 5.3, themed via CSS custom properties , grid, gap utilities, and form controls for free; brand identity supplied entirely through a token layer."}
  decision={"Option 3. Bootstrap's grid and utility classes ( row-cols-3 , g-1 , form-control ) replace the hand-written layout CSS. All AFJ-specific values live in one [data-bs-theme=\"afj-dark\"] block of custom properties, never hardcoded inside a component."}
  reasoning={"Bootstrap's own dark-mode and utility system does the layout work no-code, so the team writes brand decisions once instead of re-deriving grid math per page. Routing every colour through CSS custom properties (not Bootstrap's Sass compile step) means a future move to Tailwind is a token remap, not a rewrite, the design decisions were never coupled to the framework's internals to begin with."}

## ADR: Four-theme token system

Written retroactively, since it was never formally recorded when the third and fourth themes were added, exactly the kind of gap running an ADR pass is meant to catch.

<AdrCard
  title={"Run Wine/Gold, Scrollytelling, Day, and Scrolly Day as one token set behind a singledata-themeswitch, not four separate builds"}
  status={"Accepted"}
  context={"Once a second visual system (Scrollytelling) and a light variant (Day mode, later Scrolly Day) existed, the component library needed a way to preview all four without either duplicating every component's markup four times or shipping four separate HTML files."}
  options={"1. Four separate static files , simplest to build, but doubles as four things to keep in sync every time a component changes. 2. Four CSS files swapped via <link> , one HTML file, but a page reload on every theme switch. 3. One file, four [data-theme=\"x\"] variable blocks, one JS attribute switch , instant switching, single source of truth per component."}
  decision={"Option 3. Every component reads var(--bg-base) , var(--accent-1) , etc.; only the four theme blocks at the top of the stylesheet contain literal hex values."}
  reasoning={"A reviewer can confirm a component works in all four themes by clicking a button, not by opening four files and comparing them by eye. It also made the Scrolly Day theme trivial to add later, a fifth block, zero component changes, which is the real test of whether the token architecture was right."}
/>
## ADR: Remove the calendar from class booking

A genuine product decision from the build call, not just a UI pattern choice, worth recording as an ADR since "why isn't there a calendar" is exactly the kind of question a new team member would otherwise have to ask around about.

<AdrCard
  title={"Replace the calendar/date-picker booking flow with direct expand-to-book on a fixed weekly class list"}
  status={"Accepted"}
  context={"The original booking flow used a calendar and date picker. AFJ Cardiff's actual classes run on a fixed weekly schedule (same day, same time, every week) rather than one-off bookable slots, so a calendar was solving a scheduling problem the business doesn't have."}
  options={"1. Keep the calendar , familiar pattern, but adds a date-selection step for a schedule that never actually changes week to week. 2. Remove the calendar; list classes with an inline \"Book class\" button , matches how the schedule actually works."}
  decision={"Option 2. Each class card expands on tap to reveal its description and a direct \"Book class\" button, no date picker, no calendar widget."}
  reasoning={"Removing a whole interaction step reduces booking friction directly, and it removes a UI component (calendar) that has to be built, tested, and kept accessible for a scheduling flexibility the business doesn't use. Simpler UI, fewer edge cases, faster booking, recorded here so nobody re-adds a calendar later without knowing this was already tried and deliberately removed."}

## ADR: Adopt Wikipedia's AI-writing checklist as the style standard

Written after the fact, inferred from why the em-dash audit happened at all: a subjective request to "make it sound less AI" has no way to be checked or closed out. This ADR names the actual reasoning.

<AdrCard
  title={"Use Wikipedia's published \"Signs of AI writing\" guide as an external, checkable editing standard for this site's prose"}
  status={"Accepted"}
  context={"Documentation written collaboratively with an AI assistant risks reading as generic or formulaic, but \"sound more human\" is not a testable instruction, and every reviewer would apply it differently."}
  options={"1. Manual proofreading on a case-by-case basis , inconsistent, depends entirely on whoever's reviewing that day. 2. Ban specific words outright (delve, tapestry, and so on), treats the symptom, not the pattern. 3. Adopt WikiProject AI Cleanup's published checklist , a maintained, external, specific list of patterns, checkable the same way every time."}
  decision={"Option 3. The actual audit run against this page checked for negative parallelism, stock AI vocabulary, and em-dash density, the three most concrete, checkable items on the list."}
  reasoning={"An external, published standard means the bar doesn't move depending on who's reading. It also means the audit is repeatable: the same checklist can be re-run after future edits without reinventing the criteria each time."}

## ADR: Search filters content, category chips filter the sidebar

Inferred directly from a correction mid-build: search and category filtering were originally built to both touch the sidebar, then explicitly separated once that overlap was flagged as the wrong design.

<AdrCard
  title={"Keep search and category filtering as two separate mechanisms with one job each, combined by AND logic, not one mechanism doing both"}
  status={"Accepted"}
  context={"The first version of search filtered both the sidebar nav and the reading pane at once. Once category filter chips were added for the sidebar specifically, the two systems began fighting: searching while a category was active silently pulled sections back in from other categories."}
  options={"1. Let search override category when active, simplest to build, but silently breaks whatever category the person had deliberately chosen. 2. Let category override search , same problem in the other direction. 3. Track both states independently and require a section to satisfy both to stay visible ."}
  decision={"Option 3. A single currentCategory variable plus the live search term are combined in one applyFilters() pass, every section must pass both checks."}
  reasoning={"Two features that each silently undo the other is worse than either feature alone, since the person can no longer predict what a click or keystroke will do. Combining them by AND logic means both choices stay visibly true to what the person actually selected, tested directly against a written Gherkin scenario rather than assumed to work."}
/>

## ADR: Modern format strategy, WebP and WebM alongside originals

Written after generating the actual WebP files and measuring the real size difference, not adopted on the assumption that WebP is simply "better."

<AdrCard
  title={"Ship WebP alongside PNG for images, keep WebM-first with MP4 fallback for video, and hold off on AVIF for now"}
  status={"Accepted"}
  context={"Every image asset generated so far was PNG. PNG is safe and universal but not the smallest option for the gradient-heavy logo artwork specifically, and the existing brand-audit video already demonstrates a working multi-format delivery pattern worth extending to images."}
  options={"1. PNG only , safest compatibility, but leaves real, measurable file-size savings on the table for no reason. 2. Replace PNG with WebP entirely , smaller files, but breaks on the handful of older email clients and tools that still can't open WebP. 3. Ship WebP alongside the existing PNGs , using <picture> so the browser picks WebP when it can and falls back to PNG when it can't."}
  decision={"Option 3, for images. For video, keep the existing WebM-first, OGV, MP4-fallback pattern the Brand Audit's own logo assembly video already uses, don't reinvent it. AVIF was considered and set aside for now: real compression gains over WebP, but support isn't universal enough yet to justify a third format on top of two."}
  reasoning={"Measured, not assumed: converting the actual logo files produced real savings of 76\u201386% (81,849 bytes down to 11,596 for the 512px logo, for example). That's large enough to matter for page load, but only worth taking if nothing breaks for whoever can't render WebP, hence shipping both rather than replacing one with the other."}

## ADR: Site-wide contrast audit found via real headless testing, not inspection

**Context.** A screenshot of the deployed live site showed body text as barely visible — nearly illegible against the dark background. Rather than guess at CSS causes, built a real audit (`scripts/audit-live-site.js`) that screenshots the actual deployed `andiekobbietks.github.io` URL across all four themes and computes real WCAG contrast ratios for every visible text node, run on GitHub's CI where headless Chromium is available (this sandbox can't run one).

**Options considered.**
1. Read the CSS and guess at the cause.
2. Build a real audit that measures actual computed colors against actual computed backgrounds on the live deployed site, not the local build.

**Decision.** Option 2. Guessing from CSS alone had already produced two believable-but-wrong theories before the audit ran and gave real numbers.

**Reasoning.** The real cause was two separate bugs the audit's own numbers pinpointed precisely:
- Bootstrap 5's `body { color: var(--bs-body-color) }` was never themed globally (only inside `.form-control`), so Bootstrap's hardcoded light-mode default (`#212529`) silently won everywhere — measured at 1.26:1 against the dark page background, affecting every heading and paragraph on every page.
- The Day/Scrolly Day themes correctly lightened page content, but the sidebar/navbar chrome is governed by a *separate* `data-afj-mode` attribute that only the independent day/night switch touches — nothing set it when picking a theme via the four-theme buttons, so the sidebar stayed dark-styled against a now-light page. Measured at 1.07:1 on real sidebar links.

Both fixed and re-verified with the same audit script against the redeployed site — not assumed fixed from reading the diff. A third, smaller issue (Infima's `.navbar--dark` locally redefining `--ifm-menu-color` through a cascade path not fully traced) was fixed by targeting the actual `.menu__link`/`.navbar__link` elements directly rather than continuing to chase the variable cascade. Remaining smaller contrast gaps (the active breadcrumb pill, one un-migrated navbar background, a stray Infima-blue link) are tracked as open, not silently left for someone to rediscover.
