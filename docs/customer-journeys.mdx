### How it got here — static markup, then a working port

The original is vanilla JS manipulating the DOM directly — global mutable variables, `classList.add`/`remove` to show/hide views, `innerHTML` string-building for the cart list:

```js
// original: static-file version, direct DOM manipulation
let cart = [];
let currentSize = 'M', currentQty = 1;

function addToCart() {
    cart.push({ name: activeProduct.name, price: activeProduct.price, size: currentSize, qty: currentQty });
    updateCartUI();
    showToast('Added to basket');
    openCart(); // side effect: also opens the drawer
}

function updateCartUI() {
    document.getElementById('cartCount').textContent = cart.length;
    const itemsEl = document.getElementById('drawerItems');
    itemsEl.innerHTML = cart.map(item => `<div class="cart-line">...</div>`).join('');
}
```

The React port replaces every one of those patterns with the equivalent state-driven idiom — same behaviour, verified against the same 14 Gherkin scenarios that already test the original (quantity floor of 1 with no ceiling, size selection reflected via `aria-pressed`, adding an item auto-opens the cart, removing the only item empties it without erroring):

```js
// React port: state drives the render, not the other way round
const [cart, setCart] = useState([]);
const [size, setSize] = useState('M');
const [qty, setQty] = useState(1);

const addToCart = () => {
  setCart((prev) => [...prev, { ...activeProduct, size, qty }]);
  showToast('Added to basket');
  setCartOpen(true); // same side effect, now just a state update
};
```

The cart list itself goes from an `innerHTML` template-string loop to a real `.map()` returning JSX — no string concatenation, no risk of a stray unescaped character breaking the markup.

## Customer journeys

Not screenshots, these are the real files, live in the preview window below. Add to basket, tap a class to book it, type an email into the subscription gate: every interaction actually runs. The fifth possible journey, the staging password gate, stays out of scope here since it's an internal dev tool a customer never sees, not a shopping flow.

Every mockup above is embedded fully inline ( srcdoc , not a linked file), this one HTML file is entirely self-contained. Copy it on its own, email it, open it from anywhere, and all six journeys still render live with no other files required. That's also why this page is heavier than it looks: it's carrying six working apps inside it, not six links to them.

**Class booking. Wine/Gold**

```text
// afj-class-booking-wine.html, expand-to-book, no calendar (see ADR-004)
function bookClass(id, e) {
    e.stopPropagation();
    const c = classes.find(x => x.id === id);
    if (bookings.find(b => b.id === id)) return;
    bookings.push(c);
    btn.textContent = 'Booked ✓';
    renderBookings();
    showToast(`Booked, ${c.name}`);
}
```

**Course subscription. Wine/Gold**

```text
// afj-course-subscription-wine.html, email gate, real validation
function submitEmail() {
    const input = document.getElementById('emailInput');
    const isValid = /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(input.value.trim());
    if (!isValid) { showToast('Enter a valid email address'); return; }
    showToast('Early access confirmed');
    showView('plans');
}
```
