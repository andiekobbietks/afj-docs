How to actually use this site

Every component and every full mockup on this site works the same way: a live, interactive preview sits above, and the real, working code sits below it in a copyable block. You don't need to be a programmer to use it, see the button you want, copy the code underneath it, paste it where you need it. That's the entire workflow, start to finish.

This pattern isn't invented here, it's the same one Bootstrap's own documentation site uses (a live example, then the exact markup that produced it, copy-paste ready), which itself traces back to CodePen's live-preview-plus-code convention. Following that lineage on purpose means anyone who's ever copied a snippet from Bootstrap's docs already knows how to use this site without being taught, a designer, a marketer, or Plamedi can lift a component the same way a developer would, no separate "translate this for non-engineers" step required.

About this site

AFJ Cardiff is Wales' first dedicated African dance organisation, founded by Plamedi Santima-Akiso. This site is the internal component library and decision record for AFJ Cardiff's digital properties, the merch shop, class booking, course subscriptions, and the pre-launch scrollytelling teaser.

Three groups use this site for different reasons: engineers copy component markup and read the ADR log before making a change; designers and marketing check that a colour, radius, or piece of copy is on-brand before it ships; and anyone new to the project uses Start Here and the Glossary to get oriented without needing to ask. Nothing here is customer-facing, that's why the repo should stay private for now.

Glossary

Every piece of jargon used anywhere on this site, defined once here. Dotted-underline terms throughout the site link back to this list.

A–Z
ADR (Architecture Decision Record)
A short note capturing one decision: the context, the options considered, what was decided, and why. See the "What is an ADR" section for the full explanation.
Bootstrap
The open-source CSS framework this whole commerce build sits on, supplies the grid, gap utilities, and form controls so AFJ only has to define its own colours and type, not layout math.
Breakpoint
The screen width at which a layout changes, e.g. 500px, where a 4-column gallery collapses to 2 columns.
Design token
A named value (a colour, a radius, a font) stored as a CSS custom property instead of hardcoded, so changing it in one place updates it everywhere it's used.
Gradient clip / background-clip: text
A CSS technique that fills text with a gradient instead of a flat colour, used on every headline in the Scrollytelling theme.
IoU (Intersection over Union)
A computed shape-similarity score: how much two shapes overlap divided by their combined area. Used to measure which font actually matches the real logo letterforms, rather than guessing.
Lottie
A lightweight animation file format (JSON-based vector animation) that plays via a single web component, used instead of video for small UI motion like success checkmarks.
Pixel-sampling
Reading the actual colour value directly off an image's pixels in code, rather than estimating a hex code by eye.
Scroll-scrubbing
Tying an animation's state directly to scroll position, so the user's own scrolling, not autoplay or a click, drives the motion forward and backward.
Theme token / CSS custom property
A variable defined with --name: value in CSS, read anywhere in the stylesheet with var(--name), and swappable at runtime by changing a data-theme attribute.
Tonal scale
A set of shades derived from one base colour (darker for pressed states, lighter for text-on-fill) so an interface never needs a second, unrelated colour for interaction states.

How this fits the live site

This library is a reference, not the live site itself. AFJ Cardiff's actual production site, custom CMS, merch shop with Stripe, and a planned subscription courses area, is being built separately. This section is the honest status of that build and how the two connect, current as of the last review.

Current status

The full build is continuing rather than being replaced with a rushed placeholder, that decision was made deliberately after external feedback flagged the absence of a live site as a credibility gap with prospective partners. A fast landing-page alternative was considered and kept on the table as a bridge, not adopted as a substitute: the interim public presence is a tidied Linktree and a Canva one-pager, not a stand-in website.

Trade-off, stated plainly: this preserves brand quality control at the cost of a continued short-term credibility gap, and every week it slips, more of AFJ Cardiff's growing content sits undiscoverable behind an interim link page instead of a real site. That's a known, accepted cost, not an oversight.

What this library is for, once the build resumes

This is the design-system reference the real build should implement against, tokens, components, both themes, and the commerce and booking flows already mocked up end to end. It exists so the actual site doesn't have to make brand and component decisions from scratch mid-build, they've already been made and tested here.

Launch checklist this library doesn't cover, worth flagging now

A handful of legal and compliance requirements attach the moment the real site goes live, none of which apply yet, but all of which belong on the go-live checklist rather than being discovered after launch: a cookie notice and website terms, a privacy notice, and accessibility consideration under the Equality Act 2010 given AFJ Cardiff is a public-facing arts organisation. Worth noting separately: a UK-only site is still reachable by EU visitors by default, so the EU's cookie-consent standard applies in practice even though AFJ Cardiff itself only operates in the UK. None of this blocks using this component library now, it's simply not what this library documents.

Verified by computation, not by eye

The uploaded master logo file was cropped in Python (row/column pixel-density profiling, not manual selection), converted to binary silhouettes, and compared against real open-source font files pulled from the Google Fonts GitHub repo using Intersection-over-Union shape overlap:

Archivo BlackIoU 0.632

Montserrat Black's win independently confirms the documented recommendation above, this time by measurement rather than by eye. The 5-point gradient was also re-sampled directly from the uploaded file: the middle three stops landed within 1–3 RGB points of the documented values, the edge stops within 11–15 (likely compression on the uploaded copy), close enough to confirm the documented gradient is genuinely sampled, not invented.

For graphic designers

What this site actually has for you, beyond the component library: real source material and the constraints already discovered while working with it.

RGB vs CMYK, already flagged in the Brand Audit

The documented gradient (gold to copper to wine) was sampled directly from screen pixels, so it's an RGB value set. The Brand Audit itself is explicit about the consequence for print: those values will need careful ink calibration to CMYK before going on paper or fabric, or the warm gold and copper tones risk reading duller and colder than they do on screen. Treat any merch, print, or signage proof as the real test, not the on-screen hex codes.

Minimum size and clear space

Generating the actual favicon set surfaced a real legibility limit: at 32px and below, the Cardiff script and the interlocking continent outline stop reading as anything other than noise, the AFJ letters are the only element that survives at that size. The 180px apple-touch-icon still holds up. Practical rule derived from that: anywhere the mark will render under roughly 40px (favicons, app icons, small social avatars), use an AFJ-letters-only reduction, not the full lockup shrunk down.

Clear space: since no minimum clear-space ratio has been formally specified yet, treat the continent outline's own bounding box as the working clear-space boundary until a proper spec exists, don't crop tighter than the map's own edges.

Redraw source material

The Real Logo Imagery section above has the actual close-up crops (letter A, letter F, the Cardiff script isolated from the J, the weave crossing) pulled from the master file, computed by pixel-density profiling rather than eyeballed, useful as literal tracing reference for any vector redraw. The Typeface Families section has the computed font matches with real IoU shape-overlap scores, and the honest caveat that none of them is a guaranteed pixel match, so hand-tracing the letterforms is the safer route for anything that needs to be pixel-exact.

For videographers

What's actually relevant here for motion and video work specifically, not a generic style guide repurposed.

The existing logo assembly video

The Brand Audit already has a real motion asset: the Dynamic Logo Assembly Walkthrough, a looping animation showing the layer order the mark is built from (base letters, rear continent outline, front weave crossing, Cardiff script, tagline). It's delivered in three formats for browser compatibility, WebM (VP9), OGV (Theora), and MP4 (H.264), which is a reasonable real-world reference point for what formats to deliver in for web use generally: a modern efficient codec plus a universal MP4 fallback.

Motion guidance already documented, see "Where Lottie fits"

The one hard rule already established for motion elsewhere on this site applies here too: loop ambient or idle animation only, never loop a success or confirmation moment, it should play once and land, or it stops reading as confirmation and starts reading as decoration. If cutting a sting or bumper that ends on the logo lockup, that's a "confirmation" moment in this sense, let it settle rather than looping.

Framing, by which design system the footage is supporting

The Scrollytelling system was built deliberately cinematic, near-black canvas, wide negative space, slow reveals, so footage cut for that context (the pre-launch teaser, brand storytelling) reads correctly in a widescreen, letterboxed, patient pace. The Wine/Gold system is the transactional, calmer register, footage supporting booking or product flows should be cut tighter and more direct, closer to a product demo than a mood piece. See Compare & Contrast for the fuller reasoning behind why the two registers exist at all.

Colour note: same RGB-vs-delivery-format caveat as print applies here in spirit, screen-sampled hex values may need grading adjustment depending on the final delivery platform's colour handling, confirm on the actual delivery screen or platform rather than trusting the hex code alone.

Search

Rule of thumb: an icon never appears without a text label nearby, except inside a button whose action is already obvious from context (a lone × to dismiss a toast, a search glass inside a search field). 6. Tonal scale explainer

How the seven derived shades of the base accent actually get used across UI states. This table is token-driven, it reads --accent-2 and derives the seven stops from it, so it's the wine scale in Wine/Gold, the orange→red scale in Scrollytelling, and the same wine scale again (just on beige) in Day mode. One rule, three renders.

Preview
Darkest −40%Pressed / active button backgrounds, deep panel accents
Darker −30%Card borders and dividers on dark surfaces
Dark −20%Hover state on primary buttons and links
BasePrimary buttons, links, focus indicators, the one colour that carries meaning
Light +15%Secondary badges, tag fills on light surfaces
Lighter +25%Icon fills and decorative accents against dark backgrounds
Lightest +40%Text on dark fills, hover text on near-black panels

The rule never changes between themes: no second hue for interaction states. Every hover, pressed, disabled, or accent moment pulls from this same scale, darker for "pressed down", lighter for "on dark" or "text on fill". Wine/Gold and Day mode breathe through the wine hue; Scrollytelling breathes through the orange→red gradient. Neither theme ever mixes in an unrelated colour.

Wine/Gold vs Scrollytelling

Two design systems living in one codebase only works if they're not actually competing for the same job. They aren't, each one owns a different moment in the visitor's relationship with AFJ Cardiff, and the palette and typography choices follow directly from that.

Palette & typography, side by side

Wine / Gold

Transactional & operational

Montserrat, sentence case

Calm, legible, gets out of the way of the task

Scrollytelling

Narrative & discovery

Space Grotesk, uppercase

Deliberate, cinematic, wants to be looked at

How they complement rather than compete. They never render in the same moment. Wine/Gold is calm on purpose because it shows up once someone already trusts the brand and wants to complete a task, booking, checkout, browsing classes. Scrollytelling is loud on purpose because it shows up before that trust exists, when the job is to earn attention and explain the culture behind the brand, not process a transaction. A visitor who lands on the scrollytelling teaser and later becomes a paying member should feel like they moved from "being told a story" to "being trusted to just get on with it", that shift in register is the two palettes doing their job.

Gherkin, what Wine/Gold serves
# Wine/Gold: transactional, operational, trust-first surfaces Feature: Booking and commerce Scenario: Returning member books a weekly class Given a visitor already trusts the brand and knows what they want When they open the class list Then calm Montserrat type and a quiet wine/gold palette let the booking action be the only thing competing for attention Scenario: Customer completes a merch checkout Given a shopper has added items to the basket When they reach the checkout modal Then the interface stays legible and low-drama on purpose so nothing about the visual design creates hesitation at the point of payment
Gherkin, what Scrollytelling serves
# Scrollytelling: narrative, discovery, pre-trust surfaces Feature: First impressions and cultural storytelling Scenario: Cold visitor lands from an Instagram bio link before launch Given the visitor has never heard of AFJ Cardiff When they land on the scrollytelling page Then the gradient headline and grid overlay signal something worth paying attention to, before any commerce is even shown Scenario: Visitor needs to understand what Azonto actually looks like Given text alone cannot convey a dance movement When the visitor scrolls through the pinned story cards Then the cinematic red/orange system and Space Grotesk headlines carry the emotional weight text alone could not

Put simply: if a screen is asking someone to do something, it should look like Wine/Gold. If a screen is asking someone to feel something, it should look like Scrollytelling. Neither palette is "the real brand" and the other a variant, they're two tools built for two different jobs the same organisation genuinely has.

What is an ADR

An Architecture Decision Record is a short note that captures one decision, not a formal sign-off, not a long report, just a record anyone can read later and understand why something was built the way it was. The format below follows the UK Government Digital Service's own ADR framework, introduced by their Enterprise Architecture team to help teams across government make and record decisions consistently.

Source

The Architecture Decision Record (ADR) Framework: making better technology decisions across the public sector

Posted by Nabil Nazar, Principal Enterprise Architect, GDS, introduces the ADR framework GDS built to help teams document decisions consistently, reduce duplicated work, and keep context when people move on.
What the framework actually says

Straight from the source: an ADR documents four things, the context for the decision, the options considered, the decision made, and the reasoning behind it. GDS built the framework in two parts: guidance on when and how to use one, and a standard template for recording it, deliberately simple enough that "even someone who had no prior knowledge of ADRs could understand what was needed."

In government, ADRs aren't mandatory for everyday team decisions, but they are required for anything brought before the Technical Design Council, the body of Chief Technology Officers who sign off on decisions that affect shared platforms. That's a useful yardstick even outside government: write one when a decision is big enough that someone else would reasonably ask "wait, why did we do it that way?"

GDS's own early testing found ADRs help teams clarify their thinking and spot risks early, onboard new people faster, collaborate better across disciplines, and build a reusable record, but only if writing one becomes a normal habit, not a box-ticking exercise done after the fact.

Why it matters to non-engineers, too

For engineering

Stops the same debate happening twice. When someone asks "why don't we just use Tailwind," the ADR already has the answer, with the alternatives that were weighed and rejected on record.

For marketing / brand

Explains why a visual choice (like the wine-to-gold gradient) is locked in as policy, not a developer's personal preference, so brand reviews reference the decision instead of relitigating the colour.

For anyone joining later, including Plamedi

Ramp-up without asking around. Reading four short paragraphs (what was the situation, what were the choices, what got picked, why) explains a decision faster than a meeting would, and the reasoning is written down instead of living in one person's head or a deleted Slack thread.

Write your own, no technical background needed

Anyone can write an ADR, it's four short paragraphs, not code. If you've made a decision about the brand, the merch pricing, or how a class gets booked and you can imagine someone asking "wait, why did we do it that way?", write one. Fill in these four blanks in plain English:

1. ContextWhat was the situation? What problem were we actually trying to solve?
2. OptionsWhat were the realistic choices? Two or three is normal, include the one we didn't pick and why it was on the table.
3. DecisionWhich option did we actually go with? One or two sentences, no hedging.
4. ReasoningWhy that one? This is the part future-you (or future-Plamedi) will actually come back to read.

A worked example, non-technical: Context, we needed a name for the beige/light look of the site. Options, "Light mode" (generic), "Daytime" (too casual), "Day mode" (matches the existing dark/day pairing already used elsewhere). Decision. Day mode. Reasoning, keeps the naming pattern consistent with the dark theme so nobody has to learn two different naming systems. That's a complete, valid ADR, nothing above required opening a code editor.

How this works once the site becomes Docusaurus

Right now every ADR on this page is a hand-built block inside one big HTML file. Once this becomes a proper Docusaurus site (see Start Here for that plan), each ADR becomes its own small file instead, much easier for a non-developer to add to, since it's closer to writing a Word document than writing code:

  1. Copy the existing /adr/template.md file (four headings already in it: Context, Options, Decision, Reasoning)
  2. Rename it to the next number, adr-005-your-topic.md
  3. Fill in the four sections in plain English, same as the worked example above
  4. Save it. Docusaurus automatically adds it to the sidebar and gives it its own page, no HTML or CSS involved

That's the whole workflow. No pull requests to review CSS, no risk of breaking another section by editing the wrong <div>, which is exactly why moving the ADR log specifically (even before the rest of the site) to Docusaurus is worth prioritising: it's the part non-engineers are most likely to want to contribute to directly.

ADR: Bootstrap as the theming framework

The actual ADR behind why every component on this page is built on Bootstrap 5.3 rather than a bespoke CSS system or Tailwind, following the context → options → decision → reasoning structure.

ADR-002
Accepted

Adopt Bootstrap 5.3 as the base component framework, themed via CSS custom properties

Context

AFJ Cardiff's original site used hand-written CSS classes (.afj-btn-hero, .afj-feature-card, etc.) styled with hardcoded hex values. Every new page meant re-solving layout, responsive behaviour, and accessibility from scratch. The commerce build (product grid, cart, checkout) needed a faster, more consistent base without losing the brand's specific look.

Options considered

1. Keep hand-rolled CSS, full control, but every grid, gap, and responsive rule gets rebuilt per page.
2. Adopt Tailwind, fast utility authoring, but a compiler step and a different mental model from the team's existing CSS.
3. Adopt Bootstrap 5.3, themed via CSS custom properties, grid, gap utilities, and form controls for free; brand identity supplied entirely through a token layer.

Decision

Option 3. Bootstrap's grid and utility classes (row-cols-3, g-1, form-control) replace the hand-written layout CSS. All AFJ-specific values live in one [data-bs-theme="afj-dark"] block of custom properties, never hardcoded inside a component.

Reasoning

Bootstrap's own dark-mode and utility system does the layout work no-code, so the team writes brand decisions once instead of re-deriving grid math per page. Routing every colour through CSS custom properties (not Bootstrap's Sass compile step) means a future move to Tailwind is a token remap, not a rewrite, the design decisions were never coupled to the framework's internals to begin with.

ADR: Four-theme token system

Written retroactively, since it was never formally recorded when the third and fourth themes were added, exactly the kind of gap running an ADR pass is meant to catch.

ADR-003
Accepted

Run Wine/Gold, Scrollytelling, Day, and Scrolly Day as one token set behind a single data-theme switch, not four separate builds

Context

Once a second visual system (Scrollytelling) and a light variant (Day mode, later Scrolly Day) existed, the component library needed a way to preview all four without either duplicating every component's markup four times or shipping four separate HTML files.

Options considered

1. Four separate static files, simplest to build, but doubles as four things to keep in sync every time a component changes.
2. Four CSS files swapped via <link>, one HTML file, but a page reload on every theme switch.
3. One file, four [data-theme="x"] variable blocks, one JS attribute switch, instant switching, single source of truth per component.

Decision

Option 3. Every component reads var(--bg-base), var(--accent-1), etc.; only the four theme blocks at the top of the stylesheet contain literal hex values.

Reasoning

A reviewer can confirm a component works in all four themes by clicking a button, not by opening four files and comparing them by eye. It also made the Scrolly Day theme trivial to add later, a fifth block, zero component changes, which is the real test of whether the token architecture was right.

ADR: Remove the calendar from class booking

A genuine product decision from the build call, not just a UI pattern choice, worth recording as an ADR since "why isn't there a calendar" is exactly the kind of question a new team member would otherwise have to ask around about.

ADR-004
Accepted

Replace the calendar/date-picker booking flow with direct expand-to-book on a fixed weekly class list

Context

The original booking flow used a calendar and date picker. AFJ Cardiff's actual classes run on a fixed weekly schedule (same day, same time, every week) rather than one-off bookable slots, so a calendar was solving a scheduling problem the business doesn't have.

Options considered

1. Keep the calendar, familiar pattern, but adds a date-selection step for a schedule that never actually changes week to week.
2. Remove the calendar; list classes with an inline "Book class" button, matches how the schedule actually works.

Decision

Option 2. Each class card expands on tap to reveal its description and a direct "Book class" button, no date picker, no calendar widget.

Reasoning

Removing a whole interaction step reduces booking friction directly, and it removes a UI component (calendar) that has to be built, tested, and kept accessible for a scheduling flexibility the business doesn't use. Simpler UI, fewer edge cases, faster booking, recorded here so nobody re-adds a calendar later without knowing this was already tried and deliberately removed.

ADR: Adopt Wikipedia's AI-writing checklist as the style standard

Written after the fact, inferred from why the em-dash audit happened at all: a subjective request to "make it sound less AI" has no way to be checked or closed out. This ADR names the actual reasoning.

ADR-005
Accepted

Use Wikipedia's published "Signs of AI writing" guide as an external, checkable editing standard for this site's prose

Context

Documentation written collaboratively with an AI assistant risks reading as generic or formulaic, but "sound more human" is not a testable instruction, and every reviewer would apply it differently.

Options considered

1. Manual proofreading on a case-by-case basis, inconsistent, depends entirely on whoever's reviewing that day.
2. Ban specific words outright (delve, tapestry, and so on), treats the symptom, not the pattern.
3. Adopt WikiProject AI Cleanup's published checklist, a maintained, external, specific list of patterns, checkable the same way every time.

Decision

Option 3. The actual audit run against this page checked for negative parallelism, stock AI vocabulary, and em-dash density, the three most concrete, checkable items on the list.

Reasoning

An external, published standard means the bar doesn't move depending on who's reading. It also means the audit is repeatable: the same checklist can be re-run after future edits without reinventing the criteria each time.

ADR: Docusaurus-pattern mobile navigation

Inferred from the actual request ("fully mobile responsive like Docusaurus") rather than a generic responsive redesign brief.

ADR-006
Accepted

Collapse the sidebar into a slide-in drawer behind a hamburger button below 900px, matching Docusaurus's own mobile pattern exactly

Context

This page's 220px fixed sidebar plus a wide reading pane works on desktop but has no way to fit a narrow phone viewport without a different navigation model, and this doc is explicitly planned to become a real Docusaurus site later.

Options considered

1. Stack the sidebar above the content on mobile, pushes all reading content below a long list of links.
2. Invent a custom mobile nav pattern, more design work, and one more pattern for a future Docusaurus migration to un-learn.
3. Copy Docusaurus's own drawer pattern (hamburger, slide-in, dimmed overlay, auto-close on tap) now.

Decision

Option 3. The mobile nav behaviour is deliberately identical to what Docusaurus ships by default.

Reasoning

Since the eventual destination is a real Docusaurus site, building the same interaction pattern now means the mobile experience doesn't change shape twice, once for this HTML version and again on migration. It also means anyone testing this page on mobile today is already testing the future site's actual navigation model.

ADR: Search filters content, category chips filter the sidebar

Inferred directly from a correction mid-build: search and category filtering were originally built to both touch the sidebar, then explicitly separated once that overlap was flagged as the wrong design.

ADR-007
Accepted

Keep search and category filtering as two separate mechanisms with one job each, combined by AND logic, not one mechanism doing both

Context

The first version of search filtered both the sidebar nav and the reading pane at once. Once category filter chips were added for the sidebar specifically, the two systems began fighting: searching while a category was active silently pulled sections back in from other categories.

Options considered

1. Let search override category when active, simplest to build, but silently breaks whatever category the person had deliberately chosen.
2. Let category override search, same problem in the other direction.
3. Track both states independently and require a section to satisfy both to stay visible.

Decision

Option 3. A single currentCategory variable plus the live search term are combined in one applyFilters() pass, every section must pass both checks.

Reasoning

Two features that each silently undo the other is worse than either feature alone, since the person can no longer predict what a click or keystroke will do. Combining them by AND logic means both choices stay visibly true to what the person actually selected, tested directly against a written Gherkin scenario rather than assumed to work.

ADR: Real generated assets under a Docusaurus-matching /static folder

Inferred from the explicit requirement that downloadable assets "doesn't break" the eventual Docusaurus port.

ADR-008
Accepted

Generate the favicon set, fonts, and icons as real files under /static, matching Docusaurus's own static-asset convention exactly, rather than describing them or inventing a different folder layout

Context

A favicon and font set were needed now, for this HTML doc, but also needed to survive an eventual move to Docusaurus without requiring rework.

Options considered

1. Describe the required assets in prose without generating them, cheapest now, but produces nothing usable.
2. Generate them in a convenient ad hoc folder, usable now, but requires renaming and restructuring on migration.
3. Generate them directly under /static/img and /static/fonts, Docusaurus's own default convention.

Decision

Option 3. Favicon set generated from the real uploaded master logo at every standard size, fonts converted to self-hostable woff2, icons exported as individual SVGs, all under /static.

Reasoning

Matching the destination folder convention before the migration happens turns the whole asset-porting step into copying one folder, not a rework. The <link rel="icon"> tags already written into this page's own <head> reference static/img/favicon.ico for the same reason, so that markup carries over unchanged too.

ADR: Modern format strategy, WebP and WebM alongside originals

Written after generating the actual WebP files and measuring the real size difference, not adopted on the assumption that WebP is simply "better."

ADR-009
Accepted

Ship WebP alongside PNG for images, keep WebM-first with MP4 fallback for video, and hold off on AVIF for now

Context

Every image asset generated so far was PNG. PNG is safe and universal but not the smallest option for the gradient-heavy logo artwork specifically, and the existing brand-audit video already demonstrates a working multi-format delivery pattern worth extending to images.

Options considered

1. PNG only, safest compatibility, but leaves real, measurable file-size savings on the table for no reason.
2. Replace PNG with WebP entirely, smaller files, but breaks on the handful of older email clients and tools that still can't open WebP.
3. Ship WebP alongside the existing PNGs, using <picture> so the browser picks WebP when it can and falls back to PNG when it can't.

Decision

Option 3, for images. For video, keep the existing WebM-first, OGV, MP4-fallback pattern the Brand Audit's own logo assembly video already uses, don't reinvent it. AVIF was considered and set aside for now: real compression gains over WebP, but support isn't universal enough yet to justify a third format on top of two.

Reasoning

Measured, not assumed: converting the actual logo files produced real savings of 76–86% (81,849 bytes down to 11,596 for the 512px logo, for example). That's large enough to matter for page load, but only worth taking if nothing breaks for whoever can't render WebP, hence shipping both rather than replacing one with the other.

ADR: Continue the full site build, no rushed placeholder

Carried over from a governance review of the live production build, not made in this library, included here because it directly determines when this design system actually gets implemented. Scoped deliberately to the sequencing decision itself, the underlying contract details live in AFJ Cardiff's legal documentation, not in a design-system reference read by designers and videographers too.

ADR-010
Accepted

Continue the proper website build rather than ship a fast placeholder, bridge the gap with an interim Linktree and Canva presence instead

Context

External feedback from a business mentor flagged the absence of a live AFJ Cardiff site as a credibility gap when approaching partners. A quick landing page was proposed as a fast fix while the real build continues.

Options considered

1. Ship a fast landing page now, closes the credibility gap immediately, kept on the table as a bridge rather than rejected outright.
2. Do nothing and wait in silence, rejected, an already-long wait shouldn't be compounded with total silence toward partners.
3. Continue the proper build, use a tidied Linktree and a Canva one-pager as the interim public face.

Decision

Option 3. The real build continues; the interim presence is explicitly a bridge, not a substitute, and is not this component library either, this library is the reference the real build should implement against once it resumes.

Reasoning

Preserves brand quality control at the cost of a continued short-term credibility gap, an accepted trade-off, not an oversight. The practical consequence for this library specifically: the components, tokens, and mocked-up commerce and booking flows documented here exist precisely so the real build doesn't have to make those decisions from scratch once it resumes, the wait is for build capacity and contract mechanics external to this document, not for design decisions that are already settled.

Why Bootstrap

ADR-002 above records the formal decision. This is the plainer version, for anyone who wants the reasoning without reading it in ADR format.

The short version

Before this library, every page had its own hand-written CSS class for every layout pattern, .afj-feature-grid, .afj-btn-hero, and so on, each solving grid and spacing problems that had already been solved a dozen times over on the previous page. Every new page meant re-deriving responsive breakpoints, gap spacing, and form-control styling from scratch.

Bootstrap gives us that layout math for free, row-cols-4, g-2, form-control, and its built-in theming system (data-bs-theme) means every one of those utility classes can still be repainted with AFJ's own colours instead of Bootstrap's defaults. So the team stopped writing layout CSS and started only writing brand decisions: what a button's gradient is, what radius a card uses, what a focus ring looks like. Everything else, the grid, the spacing scale, the responsive collapse, is inherited.

The one deliberate choice worth calling out: every AFJ colour and radius is defined as a plain CSS custom property, not run through Bootstrap's Sass build step. That's what keeps the door open to changing frameworks later without redoing the design decisions, see the next section.

Future: Tailwind

Not a decision that's been made, a note for whoever revisits this in a year, so the option is on the table with context instead of being proposed cold.

Why this might eventually be worth it

Bootstrap is the right call for where this project is now, a small number of pages, a small team, and a real need to move fast without inventing a design system from zero. Tailwind's advantage shows up at a different scale: once a project has many bespoke, one-off layouts that don't fit Bootstrap's grid-and-component vocabulary, writing utility classes directly in markup (flex gap-4 rounded-xl) tends to be faster than either fighting Bootstrap's defaults or writing new custom CSS per page.

The trigger to actually revisit this: if the commerce and booking flows grow enough bespoke, non-standard layouts that Bootstrap's component set stops being a good fit more often than it helps, that's the signal, not a fixed timeline.

The reason this migration would be cheaper than it sounds: because every AFJ colour, radius, and font choice already lives in plain CSS custom properties (see Why Bootstrap above) rather than inside Bootstrap's Sass variables, moving to Tailwind would mean remapping those same tokens into a tailwind.config.js theme block and swapping Bootstrap's grid classes for Tailwind's utility classes, not redesigning anything. The design decisions were kept independent of the framework on purpose, specifically so this door would stay open.

Roadmap

What's planned but not built yet, written down for the same reason the ADRs are: so the plan lives somewhere other than one person's memory.

Docusaurus migration

This single-file HTML doc becomes a proper Docusaurus site, each section on this page (Design tokens, Buttons, ADRs, and so on) becomes its own markdown page instead of a block inside one giant file. The biggest practical win: non-engineers can add or edit a page by writing markdown, the same way the ADR workflow already described above works. The ADR log is the first candidate to move, since it's the section non-technical contributors are most likely to want to write in directly.

Search: from keyword matching to semantic search

The search box in the sidebar right now does plain keyword matching, it only finds a page if you type a word that's literally in its title. The planned upgrade is semantic search: instead of matching exact words, the search understands what a question means and surfaces the right page even if none of the words match exactly, typing "how do I make a button" would find the Button component page even though that page's title doesn't contain the word "make."

Technically, this works by converting every page into an embedding, a numerical fingerprint of its meaning, ahead of time, then converting whatever gets typed into the search box into the same kind of fingerprint and finding the closest matches. None of that complexity is visible to whoever's searching; it just feels like the search box understands the question.

The visible change: a dropdown list of suggested pages appearing live as someone types, ranked by relevance, before they've even finished the sentence or hit enter, closer to how a modern AI assistant search bar behaves than the current type-and-filter box.

Modern formats: WebP, WebM, and why they're worth having alongside the originals

Every image on this site so far has been PNG, the safe universal default, but PNG is not the smallest option for photographic or gradient-heavy assets like the logo. Real, measured comparison generated from the actual logo files on this page, not estimated:

logo.png → logo.webp81,849 bytes → 11,596 bytes, 85.8% smaller, same visual quality
apple-touch-icon.png → .webp16,586 bytes → 3,878 bytes, 76.6% smaller
android-chrome-512x512.png → .webp81,849 bytes → 11,596 bytes, 85.8% smaller

WebP (image): smaller than PNG or JPEG at equivalent quality, supports transparency like PNG does. Supported by every current browser, but a small number of older email clients and some design tools still can't open it, which is why the PNGs stay as the fallback rather than being replaced outright:

<picture> <source srcset="logo.webp" type="image/webp"> <img src="logo.png" alt="AFJ Cardiff"> </picture> /* browser picks WebP if it can, silently falls back to PNG if not */

WebM (video): already in real use, the Brand Audit's Dynamic Logo Assembly video is delivered as WebM (VP9 codec) first, with OGV and MP4 fallbacks for older browsers, exactly the same fallback logic as WebP/PNG above, smallest efficient format first, universal format as the safety net. AVIF is worth knowing about as the next format after WebP (often smaller again), but support is less universal as of this writing, so it's not adopted here yet, WebP is the current sweet spot of size savings versus compatibility.

Why this specific folder structure

Docusaurus serves anything placed in a project's /static folder directly, unprocessed, at the site root, and specifically expects a favicon at static/img/favicon.ico, referenced in docusaurus.config.js as favicon: 'img/favicon.ico'. Matching that path now, before the migration happens, means copying this /static folder into a new Docusaurus project's root is the entire favicon and font migration step, nothing to rename, nothing to reconfigure beyond that one config line.

The <link rel="icon"> tags in this page's own <head> already point at static/img/favicon.ico for the same reason, so the exact same head markup carries over unchanged too.
